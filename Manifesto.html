02 · Where I come from

I grew up obsessed with how things fit together.

Not just how code runs — how systems run. How a single automation at the right place in a workflow can cut 30% of someone's manual work. How a well-structured API unlocks an entire product idea. How the gap between "it works" and "it works well" is usually a design decision, not a technical one.

That obsession turned into a habit of building before I felt ready. It's the only way I know how to learn.

03 · What I've shipped

Small number. Real impact.

Six projects and one real-world deployment. Each one taught me something the next one needed. Ordered by what I shipped most recently.

/ 01

Internship Intelligence Dashboard ›

A full agentic pipeline with daily job board scraping via Apify, SQLite storage, TanStack Query frontend, and stack-match scoring against my own resume. Built it because I needed it. That's the best reason to build anything.

ApifySQLiteTanStack
ProblemScrolling job boards daily is noisy. Most listings aren't a stack match; relevance decays fast.
ApproachApify actors pull listings on a cron. SQLite normalizes them. A scoring pass matches tokens from my resume against each JD and ranks. TanStack Query hydrates the dashboard.
Architecture
┌─ Apify ─┬─ SQLite ─┬─ React UI
│ crons  │ history │ (TanStack)
│ scrape │ scoring │ filters
└────────┘─────────┘
Taught meScraping + scoring is a solved primitive. The real work is deciding what signal matters.
/ 02

AI Workout Tracker ›

Full-stack with Spring Boot, Java 21, SQLite via JPA, and a rule-based coaching engine on the frontend — because fitness apps are commoditized but building one teaches you what real-world data modeling looks like.

Spring BootJava 21JPA
ProblemFitness apps track what you did. Almost none close the loop on what to do next.
ApproachSpring Boot REST API, JPA over SQLite, normalized workout/set/exercise schema. A rule engine on the client suggests next sets based on volume + progression history.
Taught meReal data modeling isn't glamorous. Getting the entity relationships right is 70% of the product.
/ 03

URL Shortener + Analytics ›

Full-stack URL shortener with click analytics and a React dashboard — because "link shortener" is boring, but building one from scratch teaches you routing, auth, database design, and frontend state in a single weekend.

ReactNodeExpress
ProblemA toy problem, but every boring primitive inside it is load-bearing in real products.
ApproachExpress + nanoid for slugs, middleware logs every hit, Chart.js dashboard for click history by day / referrer.
Taught meYou learn a stack end to end by shipping one mundane app, not ten flashy half-apps.
/ 04

Inventory & Sales Tracker ›

A desktop inventory and sales tracker in C/C++, Qt, and SQLite — MVC architecture, real data persistence, a UI someone could actually sit down and use.

C++QtSQLite

04 · How I think about AI

Not chatbots. Not wrappers. Systems.

I'm not chasing the trend. I'm building toward a specific thing: AI systems that do real work inside real products.

The kind where an agent ingests a job board, scores the fit, drafts a tailored email, and tracks the response — all without you touching it. The kind where a Shopify store owner wakes up and their abandoned cart sequence, product descriptions, and review requests ran overnight. The kind where AI earns its place in the stack because it makes the workflow actually better.

I've been building AI-assisted workflows since before it was obvious everyone should. Structured prompting, screenshot-comparison iteration, Claude Code pipelines, MCP configurations — not as a party trick, but as a repeatable engineering methodology.

→  AI becomes powerful when it's part of a system. That's the sentence I'm building my career around.
